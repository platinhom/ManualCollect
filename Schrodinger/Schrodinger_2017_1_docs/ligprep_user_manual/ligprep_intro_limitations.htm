LigPrep Limitations

The limitations in this sections apply to the LigPrep process as a whole. Information on the limitations of individual tools is given in the sections for each tool.

Target Molecule Classes

LigPrep is intended for use with largely organic, drug-like molecules. While it may be useful for molecules outside this class of compounds it has not been developed for or extensively tested with such systems.

By default some stages of LigPrep pass over molecules containing more than 200 atoms. This limit can be adjusted using the -ma option.

Processing Problems May Eliminate Molecules From the Output Data Set

It might not be possible for the various stages in the LigPrep process to accommodate all input structures. Under such circumstances, these problematic structures might not be retained in the output structure file for that stage and thus might be absent in the final output structure file from the LigPrep process. The molecules that failed are copied to a separate file. To learn more about this issue, see the descriptions of the individual programs.

Problematic structures might also be returned in the output. These are labeled with a text field that describes the processing problems.

Processing Time Considerations

LigPrep takes about 1-2 seconds on average to process a ligand. This allows an estimate of the time needed to process a set of ligands: for example, a million ligands would take around 300 to 600 hours of processing time, or 2−4 weeks. For such large numbers of ligands, serial jobs are strongly discouraged, as the likelihood of a computer failure of some sort in this time frame is fairly high. To prevent such an occurrence, we recommend distributing LigPrep jobs over multiple processors, both to reduce the turnaround time and to take advantage of the automatic restart of failed subjobs, so that only a small amount of time is lost in the event of a failure. For more information see Distributing LigPrep Jobs. It might also be worth splitting up very large structure files using maesubset or sdsubset and running each file separately, if computer stability is an issue.
